# 기술 스택 결정서 (blog-v2 NFR)

## 결정 요약

| 영역 | 결정 | 근거 |
|---|---|---|
| 프레임워크 | Next.js 14 (App Router) 유지 | 기존 코드 자산 활용, 정적 export 지원 |
| 언어 | TypeScript strict 유지 | 타입 안정성, 유지보수성 향상 |
| 스타일 | Tailwind CSS 유지 | 반응형/디자인 일관성 확보 |
| 콘텐츠 소스 | Markdown-first + Adapter 패턴 | 빌드 안정성 및 콘텐츠 관리 단순화 |
| 빌드/배포 | GitHub Actions + AWS S3/CloudFront | 정적 사이트 운영에 적합 |
| 인프라 IaC | Terraform 유지 | 선언적 인프라 관리, 재현성 |

## 세부 결정

### 1) 콘텐츠 파이프라인
- **결정**: `src/lib/content/*` 중심으로 데이터 로더/정규화/조회 분리
- **대안**: Notion 직접 호출 중심 유지
- **선택 이유**:
  - 빌드 안정성 향상 (외부 API 의존 축소)
  - 로컬 작성/검증 워크플로우 단순화
  - 하이브리드 전환 시 어댑터만 확장하면 됨

### 2) 렌더링 전략
- **결정**: 정적 export 고정 (`output: 'export'`)
- **대안**: SSR/ISR
- **선택 이유**:
  - 현재 인프라(S3/CloudFront)와 정합성 높음
  - 운영 복잡도 및 비용 예측 용이

### 3) 성능 전략
- **결정**:
  - 이미지 자산 사전 최적화
  - 목록/상세 데이터 사전 계산
  - 메타/피드/sitemap 빌드 타임 생성
- **대안**: 런타임 계산 중심
- **선택 이유**: 정적 배포 모델에서 TTFB와 안정성 최적화

### 4) 품질 게이트
- **결정**: CI에서 `type-check`, `lint`, `build`를 필수 게이트로 사용
- **대안**: build-only 게이트
- **선택 이유**: 배포 전 타입/정적분석 오류 조기 차단

### 5) 보안 운용
- **결정**:
  - AWS 키는 GitHub Secrets로만 주입
  - S3 public access block + CloudFront 경유
  - 보안 헤더 정책 적용
- **대안**: 수동 배포/퍼블릭 S3
- **선택 이유**: 키 노출 위험 및 직접 접근 표면 최소화

## 트레이드오프
- Markdown-first는 협업 편의성은 높지만 CMS 편집 UX는 제한적일 수 있음
- 정적 export는 런타임 유연성이 낮지만 운영 안정성과 비용 예측이 우수함
- 엄격한 CI 게이트는 초기 개발 속도에 일부 부담이 있으나, 배포 품질을 크게 향상함

## 향후 재검토 시점
- 포스트 수 500건 이상 증가 시 빌드 시간/메모리 프로파일 재측정
- Notion 하이브리드 재도입 시 Source Strategy 우선순위 정책 재결정
- 트래픽 급증 시 CloudFront 캐시 정책 및 이미지 전달 전략 재조정
