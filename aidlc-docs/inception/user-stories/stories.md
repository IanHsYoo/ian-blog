# 사용자 스토리

## 개요
본 문서는 ian-blog v2 범위 확장(마크다운 우선 콘텐츠, 카테고리/태그, 포트폴리오 통합, 모바일 대응)을 위한 사용자 스토리를 정의한다.
스토리는 기능 중심으로 구성하고, 각 스토리에 페르소나를 매핑한 하이브리드 방식으로 작성한다.

---

## 스토리 구성 원칙
- **Independent**: 각 스토리는 독립적으로 구현/검증 가능
- **Negotiable**: 구현 방식은 협의를 통해 조정 가능
- **Valuable**: 사용자 가치가 명확함
- **Estimable**: 범위 추정 가능
- **Small**: 과도하게 크지 않은 단위로 분해
- **Testable**: 수용 기준으로 검증 가능

## Epic A: 콘텐츠 탐색 경험

### US-01: 홈에서 최신 포스트 탐색
**As a** 기술 블로그 독자(민수)  
**I want** 홈에서 최신 포스트를 카드 형태로 빠르게 훑어보고  
**So that** 관심 있는 글을 빠르게 선택하고 읽을 수 있다.

**페르소나 매핑**: 민수(주), 지영(보조)

**Acceptance Criteria (Given/When/Then)**
1. Given 발행된 포스트가 6개 이상일 때, When 사용자가 홈(`/`)에 접속하면, Then 최신 발행순 기준 상위 6개 카드가 노출된다.
2. Given 포스트 카드가 렌더링될 때, When 사용자가 카드를 확인하면, Then 제목/설명/발행일/카테고리/태그/읽기 시간이 표시된다.
3. Given 사용자가 "모든 포스트 보기" CTA를 클릭할 때, When 클릭 이벤트가 발생하면, Then `/blog`로 이동한다.
4. Given 모바일 화면(<= 640px)에서 접속할 때, When 카드 목록이 표시되면, Then 1열 레이아웃으로 가독성이 유지된다.
5. Given 다크모드가 활성화되어 있을 때, When 홈 카드가 표시되면, Then 텍스트 대비가 충분하고 hover/focus 상태가 식별 가능하다.

### US-02: 블로그 목록 필터 탐색
**As a** 기술 블로그 독자(민수)  
**I want** 카테고리/태그 필터로 글을 좁혀서 보고  
**So that** 필요한 주제의 글만 빠르게 찾을 수 있다.

**페르소나 매핑**: 민수(주)

**Acceptance Criteria (Given/When/Then)**
1. Given 사용자가 `/blog`에 접속했을 때, When 목록이 로드되면, Then 발행 포스트가 발행일 내림차순으로 표시된다.
2. Given 사용자가 카테고리 필터를 선택했을 때, When URL이 갱신되면, Then `?category={값}` 형태의 쿼리가 반영된다.
3. Given 사용자가 태그 필터를 선택했을 때, When URL이 갱신되면, Then `?tag={값}` 형태의 쿼리가 반영된다.
4. Given category와 tag가 동시에 선택되었을 때, When 목록이 갱신되면, Then 두 조건을 모두 만족하는 포스트만 노출된다.
5. Given 모바일 화면에서 필터 UI를 사용할 때, When 필터 패널을 열고 닫으면, Then 레이아웃 깨짐 없이 조작 가능하다.

### US-03: 카테고리/태그 전용 페이지 탐색
**As a** 기술 블로그 독자(민수)  
**I want** 카테고리 페이지와 태그 페이지에서 관련 글을 한 번에 보고  
**So that** 주제별로 깊게 탐색할 수 있다.

**페르소나 매핑**: 민수(주)

**Acceptance Criteria (Given/When/Then)**
1. Given 사용자가 `/categories/{category}`에 진입했을 때, When 페이지가 로드되면, Then 해당 카테고리 포스트만 표시된다.
2. Given 사용자가 `/tags/{tag}`에 진입했을 때, When 페이지가 로드되면, Then 해당 태그 포스트만 표시된다.
3. Given 일치하는 포스트가 없는 분류에 접근할 때, When 목록 렌더링을 시도하면, Then 빈 상태 안내 문구와 복귀 링크가 제공된다.
4. Given 카테고리/태그 페이지에서 포스트 카드를 클릭할 때, When 상세 이동이 발생하면, Then `/blog/[slug]`로 정상 이동한다.
5. Given 모바일 화면에서 긴 태그명이 존재할 때, When 태그 UI가 표시되면, Then 줄바꿈/overflow 처리로 레이아웃 손상이 없다.

## Epic B: 포스트 읽기 경험

### US-04: 마크다운 기반 포스트 상세 읽기
**As a** 기술 블로그 독자(민수)  
**I want** 마크다운 본문을 구조적으로 읽고 코드 예제를 확인하며  
**So that** 실무에 바로 적용할 수 있는 인사이트를 얻는다.

**페르소나 매핑**: 민수(주), 지영(보조)

**Acceptance Criteria (Given/When/Then)**
1. Given 사용자가 `/blog/[slug]`에 접근했을 때, When 유효한 slug라면, Then 마크다운 본문이 제목/문단/목록/이미지/인용/콜아웃 형태로 렌더링된다.
2. Given 본문에 코드 블록이 포함될 때, When 코드가 표시되면, Then 구문 강조가 적용되고 가독성 있는 폰트/간격이 유지된다.
3. Given 포스트 메타데이터가 존재할 때, When 상세 상단이 렌더링되면, Then 발행일/카테고리/태그/읽기 시간이 표시된다.
4. Given 존재하지 않는 slug 접근 시, When 데이터 조회 결과가 비어 있으면, Then 404 또는 Not Found 화면을 제공한다.
5. Given 모바일 화면에서 긴 코드 라인이 존재할 때, When 코드 블록을 표시하면, Then 가로 스크롤 또는 줄바꿈 정책으로 콘텐츠 손실 없이 읽을 수 있다.

### US-05: SEO 메타 노출
**As a** 검색 유입 사용자(민수)  
**I want** 검색 결과와 소셜 미리보기에서 글의 핵심 정보를 확인하고  
**So that** 클릭 전에 콘텐츠 적합성을 판단할 수 있다.

**페르소나 매핑**: 민수(주), Ian(보조)

**Acceptance Criteria (Given/When/Then)**
1. Given 포스트 상세 페이지가 빌드될 때, When 메타 태그가 생성되면, Then `title`과 `description`이 포스트 데이터와 일치한다.
2. Given OG 이미지가 존재할 때, When 소셜 공유 미리보기가 생성되면, Then `og:image`가 올바른 URL로 설정된다.
3. Given RSS/사이트맵 생성을 수행할 때, When 빌드가 완료되면, Then 발행 포스트 기준으로 최신 항목이 반영된다.
4. Given draft 상태 포스트가 존재할 때, When 메타/피드 생성을 수행하면, Then draft는 제외된다.

## Epic C: About/포트폴리오 경험

### US-06: About에서 작성자 소개 확인
**As a** 채용 담당자(지영)  
**I want** About 페이지에서 작성자의 핵심 소개를 빠르게 확인하고  
**So that** 후보자의 배경을 단시간에 이해할 수 있다.

**페르소나 매핑**: 지영(주), 민수(보조)

**Acceptance Criteria (Given/When/Then)**
1. Given 사용자가 `/about`에 진입했을 때, When 페이지 상단이 렌더링되면, Then 작성자 소개와 핵심 태그라인이 표시된다.
2. Given 소개 섹션이 렌더링될 때, When 모바일 화면에서 확인하면, Then 텍스트 줄길이/간격이 가독성을 유지한다.
3. Given 다크모드에서 About 페이지를 볼 때, When 섹션 간 구분이 표시되면, Then 명도 대비로 정보 블록이 식별 가능하다.

### US-07: 경력 타임라인 중심 포트폴리오 탐색
**As a** 채용 담당자(지영)  
**I want** 경력 타임라인을 시간순으로 확인하고  
**So that** 후보자의 성장 경로와 역할 변화를 빠르게 파악할 수 있다.

**페르소나 매핑**: 지영(주)

**Acceptance Criteria (Given/When/Then)**
1. Given `/about`의 경력 섹션이 렌더링될 때, When 사용자가 스크롤하면, Then 최신 경력부터 시간순으로 주요 경력 항목이 표시된다.
2. Given 각 경력 항목이 표시될 때, When 사용자가 항목을 확인하면, Then 기간/역할/조직/핵심 성과를 확인할 수 있다.
3. Given 모바일 화면에서 타임라인을 볼 때, When 세로 레이아웃이 적용되면, Then 요소 겹침 없이 순차적으로 읽을 수 있다.
4. Given 데이터가 비어 있는 항목이 있을 때, When 렌더링이 수행되면, Then 페이지 에러 없이 안전한 기본값 또는 숨김 처리가 적용된다.

### US-08: 프로젝트 목록과 기술 스택 확인
**As a** 채용 담당자(지영)  
**I want** 프로젝트 목록과 기술 스택을 확인하고 필요한 외부 링크로 이동하여  
**So that** 후보자의 실무 역량을 검증할 수 있다.

**페르소나 매핑**: 지영(주), 민수(보조)

**Acceptance Criteria (Given/When/Then)**
1. Given 프로젝트 데이터가 존재할 때, When `/about` 프로젝트 섹션이 렌더링되면, Then 프로젝트명/설명/기술스택이 카드로 노출된다.
2. Given 프로젝트에 GitHub 또는 Demo 링크가 있을 때, When 사용자가 링크를 클릭하면, Then 새 탭으로 이동한다.
3. Given 스킬셋 데이터가 존재할 때, When 스킬 섹션을 확인하면, Then 기술군 단위로 구조화된 목록을 확인할 수 있다.
4. Given 모바일 화면에서 카드가 많을 때, When 스크롤하면, Then 카드 간 간격과 터치 타겟이 유지된다.

## Epic D: 작성자 운영 경험

### US-09: 마크다운 발행 워크플로우
**As a** 블로그 작성자(Ian)  
**I want** 로컬 마크다운 파일과 프론트매터만으로 글을 발행 제어하고  
**So that** Notion 없이도 일관된 작성/배포 흐름을 유지할 수 있다.

**페르소나 매핑**: Ian(주)

**Acceptance Criteria (Given/When/Then)**
1. Given 작성자가 `content/posts/*.md` 파일을 추가/수정했을 때, When 빌드를 수행하면, Then 유효한 frontmatter를 가진 포스트가 목록/상세에 반영된다.
2. Given 포스트의 `status`가 `draft`일 때, When 사이트를 빌드하면, Then 해당 포스트는 어떤 공개 페이지에도 나타나지 않는다.
3. Given 필수 frontmatter(`title`, `slug`, `description`, `publishedDate`)가 누락될 때, When 데이터 파싱을 수행하면, Then 빌드 단계에서 식별 가능한 오류 또는 경고가 제공된다.
4. Given 동일한 `slug`가 중복될 때, When 라우트 생성 단계가 실행되면, Then 충돌을 감지하고 배포 전 확인할 수 있어야 한다.

### US-10: 하이브리드 전환 대비 데이터 레이어 분리
**As a** 블로그 작성자(Ian)  
**I want** 콘텐츠 접근 로직이 데이터 레이어에 캡슐화되어 있고  
**So that** 추후 Notion 연동 시 페이지/컴포넌트 변경을 최소화할 수 있다.

**페르소나 매핑**: Ian(주)

**Acceptance Criteria (Given/When/Then)**
1. Given 페이지 컴포넌트가 콘텐츠를 조회할 때, When 구현을 확인하면, Then 데이터 접근은 `src/lib/content/` 레이어를 통해 이뤄진다.
2. Given 콘텐츠 소스 정책이 변경될 때, When 구현 범위를 검토하면, Then 페이지/프레젠테이션 계층 수정 없이 데이터 레이어 중심으로 변경 가능하다.
3. Given 기존 Notion 연동 코드가 남아 있을 때, When 마크다운 우선 정책을 적용하면, Then 현재 런타임 경로에서 마크다운 소스가 우선 사용된다.

## Epic E: 반응형 및 접근성 품질

### US-11: 모바일 우선 반응형 품질
**As a** 모바일 방문자(민수/지영)  
**I want** 모든 주요 페이지가 모바일에서 자연스럽게 동작하고  
**So that** 디바이스와 관계없이 콘텐츠를 불편 없이 이용할 수 있다.

**페르소나 매핑**: 민수(주), 지영(주)

**Acceptance Criteria (Given/When/Then)**
1. Given 뷰포트가 `sm` 미만일 때, When 홈/블로그/상세/About를 확인하면, Then 가로 스크롤 없이 핵심 콘텐츠가 표시된다.
2. Given 태블릿(`md`) 및 데스크탑(`lg` 이상)에서 확인할 때, When 레이아웃이 변경되면, Then 컬럼 수/간격이 자연스럽게 확장된다.
3. Given 터치 환경에서 상호작용 요소를 사용할 때, When 버튼/링크를 탭하면, Then 오탭을 줄일 수 있는 크기와 간격이 유지된다.
4. Given 긴 텍스트/코드/태그가 존재할 때, When 렌더링되면, Then overflow 처리로 레이아웃 붕괴가 발생하지 않는다.

### US-12: 테마 일관성 및 가독성
**As a** 모든 방문자  
**I want** 라이트/다크 모드에서 동일한 정보 구조와 충분한 대비를 유지하고  
**So that** 선호 테마와 무관하게 안정적으로 읽을 수 있다.

**페르소나 매핑**: 민수, 지영

**Acceptance Criteria (Given/When/Then)**
1. Given 사용자가 테마 토글을 변경할 때, When 페이지를 새로고침하면, Then 선택 테마가 유지된다.
2. Given 라이트/다크 모드 각각에서 본문 텍스트를 볼 때, When 대비를 확인하면, Then 본문/보조텍스트/링크의 가독성이 확보된다.
3. Given 카드/버튼/배지 컴포넌트가 공통 사용될 때, When 테마가 전환되면, Then hover/focus/active 상태가 모두 식별 가능하다.

## 검증 메모
- 사용자 요청 기준(미니멀 디자인, 포트폴리오 통합, 마크다운 우선, 모바일 대응, 기존 기능 유지)을 모두 커버하도록 스토리를 구성함.
- 각 스토리는 수용 기준을 통해 테스트 가능한 형태로 작성함.
- 구현 순서는 별도 워크플로우 계획 단계에서 우선순위화함.
